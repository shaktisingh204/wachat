## Part A — AI computed fields (`FieldType: 'AI'`)

An AI field is a metadata-defined field whose value is **generated by an LLM
from a prompt template** that interpolates sibling fields (Attio-style
"research/summarise/classify" columns). Values are recomputed when their
inputs change, render everywhere through the RecordCell registry, and are
filterable/sortable like any other field.

### A1. Metadata model — prompt-as-config

**`src/lib/sabcrm/types.ts`**

1. Add `'AI'` to the `FieldType` union (after `'RICH_TEXT_V2'`), with a doc
   comment: `/** LLM-computed field. Config in settings.ai; value is a plain scalar in data. */`
2. Add the optional pass-through key to `FieldMetadata` (aligns the base type
   with the Rust wire shape `SabcrmFieldMetadata` already declares):

```ts
/** Type-discriminated per-field settings blob (Twenty parity; AI fields use settings.ai). */
settings?: Record<string, unknown>;
```

3. Add the typed config interface + parser (exported, used by UI + evaluator):

```ts
/** What an AI field produces. Drives coercion, display and filter operators. */
export type AiOutputType = 'TEXT' | 'NUMBER' | 'BOOLEAN' | 'SELECT' | 'RATING';

/** `settings.ai` blob on a FieldMetadata of type 'AI'. */
export interface AiFieldConfig {
  /** Prompt template; `{{fieldKey}}` tokens interpolate sibling data values. */
  prompt: string;
  /** Output coercion target. SELECT requires `field.options`. */
  outputType: AiOutputType;
  /** 'auto' = scheduler recomputes when inputs change; 'manual' = only on demand. */
  refresh: 'auto' | 'manual';
}

/** Parse a field's settings.ai defensively; null when absent/malformed. */
export function aiFieldConfig(field: FieldMetadata): AiFieldConfig | null;
```

`aiFieldConfig` returns null unless `field.type === 'AI'`,
`settings.ai.prompt` is a non-empty string, `outputType` is one of the five
literals (default `'TEXT'`), `refresh` ∈ {auto, manual} (default `'auto'`).

**No Rust change**: `rust/crates/sabcrm-objects/src/dto.rs` types
`FieldMetadata.r#type` as `String` and `settings` as `Option<Value>`; the
PATCH handler `$set`s the whole `fields` array verbatim, so `'AI'` +
`settings.ai` persist and round-trip today (verified against `dto.rs` lines
52–88 and `handlers.rs` update path). Optionally add a doc-comment mention of
`'AI'` in the dto rustdoc — no behaviour change.

### A2. Value storage envelope — keep `data.<key>` a plain scalar

To keep filtering/sorting/grouping working with **zero engine change**, the
generated value is stored as a plain scalar at `data[field.key]` (string /
number / boolean — exactly what the records crate's filter tree at
`rust/crates/sabcrm-records/src/handlers.rs` lines 302–400 compares against).
Computation metadata rides a parallel reserved key:

```jsonc
// record.data
{
  "summary": "Enterprise fintech, expanding to EU…",   // data.<fieldKey> — the scalar
  "__ai": {
    "summary": {
      "inputsHash": "a1b2…",        // sha256(prompt + JSON of source-field values), hex, 32 chars
      "computedAt": "2026-06-12T…", // RFC3339
      "status": "ready",            // "pending" | "ready" | "failed"
      "error": null                  // failure detail when status === "failed"
    }
  }
}
```

Rules:
- `__ai` is a **system namespace**: it must never appear as a column/field.
  Nothing renders it today (no field is keyed `__ai`), so no UI guard is
  strictly required, but the evaluator must only ever write
  `data.__ai.<fieldKey>` subkeys, never replace the whole `__ai` map (use
  dotted `$set` paths).
- Scheduler writes go **direct to Mongo** (`sabcrm_records` collection,
  `$set: { "data.<key>": value, "data.__ai.<key>": meta }`) and deliberately
  do NOT bump the record's top-level `updatedAt` — same rationale as the
  deal-rotting tag design in `scheduler.ts` (lines 119–127): an AI write must
  not reset the `time.elapsed` / rotting idle clocks or re-trigger
  record-change workflows.

### A3. Shared LLM helper — `src/lib/sabcrm/ai-llm.server.ts` (new)

Extract the provider ladder from `src/app/api/sabcrm/ai/route.ts` into a
reusable, **non-streaming** server helper (the route keeps its own streaming
code; do not refactor it in this task):

```ts
import 'server-only';

export interface SabcrmLlmInput {
  system: string;
  prompt: string;
  maxTokens?: number;          // default 1024
}
export type SabcrmLlmResult =
  | { ok: true; text: string }
  | { ok: false; error: string };

/** Provider ladder, mirroring /api/sabcrm/ai exactly: AI Gateway first
 *  (`generateText` from 'ai' with the route's GATEWAY_MODEL slug), then the
 *  route's two direct-provider fetch fallbacks, else
 *  { ok:false, error:'AI is not configured…' }. Reads the SAME env keys the
 *  route already reads — gateway credential (static key or OIDC token) first,
 *  then the two direct-provider keys. */
export async function generateSabcrmText(input: SabcrmLlmInput): Promise<SabcrmLlmResult>;
```

Implementation notes: gateway branch `await import('ai')` then
`generateText({ model: GATEWAY_MODEL, system, prompt })` (dynamic import like
the route's `viaGateway`); the two direct-provider branches are the route's
`viaAnthropic`/`viaOpenAI` bodies minus streaming (return the joined text).
Never throws — all failures become `{ ok:false }`.

Model-id rules:
- The **AI Gateway** is the preferred path; gateway slugs are
  `provider/model` with **dots** for version numbers (e.g.
  `anthropic/claude-sonnet-4.6`) — never hyphenate a gateway slug version.
- The direct-provider fallback branches use each provider's own native model
  id format (Anthropic's direct Messages-API ids are hyphenated by design —
  do not convert them to gateway slug form, and vice versa). They exist only
  for environments without a gateway credential; this deployment is
  self-hosted PM2, so a static gateway key or direct keys are both
  legitimate, with the gateway preferred for routing/failover/cost tracking.
- Do **not** hardcode fresh literals in the helper: hoist the route's three
  existing model constants (`GATEWAY_MODEL`, `ANTHROPIC_MODEL`,
  `OPENAI_MODEL` in `src/app/api/sabcrm/ai/route.ts` lines 44–46) into
  `ai-llm.server.ts`, export them, and refactor the route to import them —
  one place to bump models later (the OpenAI fallback constant in the route
  is dated and due a bump; do that in the shared constant, verifying current
  ids against the gateway model list rather than memory).

### A4. Evaluator core — `src/lib/sabcrm/ai-fields.server.ts` (new)

Shared by the scheduler pass (A5) and the manual action (A6).

```ts
import 'server-only';

export interface AiFieldEvalResult {
  status: 'ready' | 'failed' | 'skipped';
  value?: unknown;            // the coerced scalar written to data.<key>
  detail?: string;            // failure / skip reason
}

/** Parse {{token}} field keys out of a prompt template (regex /\{\{\s*([\w.]+)\s*\}\}/g),
 *  intersected with the object's field keys; 'updatedAt'/'createdAt' allowed. */
export function aiSourceFields(prompt: string, fieldKeys: ReadonlySet<string>): string[];

/** sha256 hex (32 chars) over JSON.stringify({ prompt, values }) where values is
 *  the ordered source-field → raw data value map. Mirrors hashKey() in
 *  src/app/actions/sabsheet-ai.actions.ts. */
export function aiInputsHash(prompt: string, values: Record<string, unknown>): string;

/** Coerce raw LLM text by outputType. SELECT matches option value OR label
 *  case-insensitively against field.options and stores the option VALUE;
 *  NUMBER → Number() (reject NaN); BOOLEAN → /^(true|yes)$/i; RATING → int 1..5.
 *  Failure → { status:'failed', detail }. */
export function coerceAiOutput(text: string, field: FieldMetadata): AiFieldEvalResult;

/** Full single-record evaluation: render prompt via the templates engine
 *  (rustServiceFetch POST /v1/sabcrm/templates/preview with
 *  { projectId, body: cfg.prompt, object, recordId } — the exact call shape of
 *  renderSequenceEmail in scheduler.ts), call generateSabcrmText with system
 *  prompt AI_FIELD_SYSTEM (below), coerce, and $set
 *  data.<key> + data.__ai.<key> directly on sabcrm_records (no updatedAt bump).
 *  Returns the result; never throws. */
export async function evaluateAiField(args: {
  db: Db;                      // from connectToDatabase()
  projectId: string;
  objectSlug: string;
  field: FieldMetadata;        // type 'AI', config pre-parsed by caller
  recordId: string;
  inputsHash: string;          // precomputed by the caller (claim key)
}): Promise<AiFieldEvalResult>;
```

System prompt constant:

```
AI_FIELD_SYSTEM =
  'You compute one CRM field value. Reply with ONLY the value — no preamble,
   no markdown, no quotes. If the inputs are insufficient, reply with the
   single word UNKNOWN.'
```

`UNKNOWN` replies → `status:'failed', detail:'model could not determine a value'`
(value untouched). Output for SELECT additionally appends the allowed option
list to the user prompt: `Allowed values: <label (value)>, …`.

### A5. Evaluation job — scheduler pass 6

Extend `src/lib/sabcrm/scheduler.ts` (cron `/api/cron/sabcrm-workflows`,
every 5 min, 300 s budget). Follow the existing pass pattern exactly
(best-effort try/catch, caps, report lines).

New constants (alongside lines 84–107):

```ts
const MAX_AI_FIELD_OBJECTS_PER_RUN = 10;   // (project, object) pairs scanned
const MAX_AI_FIELD_EVALS_PER_RUN  = 20;    // LLM calls per tick (~1–3 s each)
const MAX_RECORDS_PER_AI_FIELD    = 200;   // records scanned per field
const AI_FIELD_CLAIM_DELAY_MS     = 10 * 60_000; // crash-retry backoff (mirrors SEQUENCE_CLAIM_DELAY_MS)
const OBJECTS_COLL = 'sabcrm_objects';
```

Report: add `aiFields: AiFieldItemReport[]` to `SchedulerReport` with
`{ projectId, objectSlug, fieldKey, recordId, status: 'ran'|'skipped'|'failed', detail? }`,
fold counts into `ran`/`failed` and the summary string like every other pass.

Pass algorithm (insert as `--- 6. AI computed fields ---` before the summary
block, line ~1629):

1. **Discover**: `db.collection(OBJECTS_COLL).find({ 'fields.type': 'AI' }).limit(500)`.
   This covers custom objects AND standard-object extension docs (the
   `sabcrm_objects` collection holds both — see crate rustdoc, `handlers.rs`
   line 16). Group by `(projectId, slug)`, cap at `MAX_AI_FIELD_OBJECTS_PER_RUN`.
2. Per object, per AI field with `aiFieldConfig(field)?.refresh === 'auto'`:
   scan `sabcrm_records` `{ projectId, object, deletedAt: { $in: [null] } }`
   capped at `MAX_RECORDS_PER_AI_FIELD` (reuse the `RecordDoc` loose shape).
3. **Dirty check** per record: compute `hash = aiInputsHash(cfg.prompt, pick(data, aiSourceFields(...)))`;
   skip when `data.__ai?.[key]?.inputsHash === hash && status !== 'failed'`.
   (Failed entries with an unchanged hash are also skipped — no hot retry; a
   changed input or manual recompute clears them.)
4. **Claim** (single-winner, mirrors the sequence claim at lines 1532–1543):
   `updateOne({ _id, 'data.__ai.<key>.inputsHash': { $ne: hash } }, { $set: { 'data.__ai.<key>': { inputsHash: hash, status: 'pending', computedAt: nowIso } } })`
   — `modifiedCount === 0` ⇒ another tick claimed it; skip. (The claim writes
   the hash up front so a crashed tick leaves a `pending` row that is skipped
   until `AI_FIELD_CLAIM_DELAY_MS` has passed since its `computedAt`; encode
   that check in step 3.)
5. **Bill**: resolve the tenant owner via the existing module-local
   `resolveProjectOwner(db, projectId)` (cache per tick in a
   `Map<string, string | undefined>` like the sequences pass). Gate with
   `canUse(ownerId, 'ai_requests')` — quota exhausted ⇒ mark the meta
   `status:'failed', error:'AI quota exceeded'` and report `skipped`.
6. **Evaluate** via `evaluateAiField(...)`; on success `recordUsage({ tenantId: ownerId, feature: 'ai_requests', units: 1, idempotencyKey: 'sabcrm-ai-field:' + recordId + ':' + field.key + ':' + hash, meta: { feature: 'sabcrm', op: 'aiField', object: objectSlug } })`.
7. Stop the whole pass once `MAX_AI_FIELD_EVALS_PER_RUN` LLM calls have run.

## Part B — Natural-language filtering (NL → FilterGroup)

User types *"enterprise companies in Berlin added last month"*; an LLM emits
the canonical `FilterGroup` tree; the user **reviews it in the FilterBuilder
draft** before applying (never auto-applied — same draft/apply contract the
`FilterControl` popover already owns).

### B1. Server action — `nlToFilterTw` (same new file as A6)

```ts
export interface NlFilterResult {
  group: FilterGroup;           // validated, pruned, ready for setDraft
  /** Model notes for parts it could not express (shown muted in the popover). */
  unresolved?: string;
}
export async function nlToFilterTw(
  objectSlug: string,
  query: string,
  projectId?: string,
): Promise<ActionResult<NlFilterResult>>;
```

Pipeline:
1. `gate('view', projectId)`; reject empty/`> 500`-char queries.
2. `canUse(g.ctx.userId, 'ai_requests')` → on success later `recordUsage`
   with key `sha256('sabcrm-nl-filter:' + objectSlug + ':' + query)`
   (deterministic, mirrors `hashKey` in `sabsheet-ai.actions.ts`).
3. Field catalogue: `sabcrmObjectsApi.get(objectSlug, g.ctx.projectId)`;
   serialise each non-RELATION/non-FILE field (the `filterableFields` rule) as
   `key | label | type | options(value:label)…`.
4. `generateSabcrmText` with:
   - system: *"You translate a natural-language request into a JSON filter for
     a CRM object. Reply with ONLY minified JSON, no markdown."*
   - prompt: the catalogue, today's date (`new Date().toISOString().slice(0,10)`
     so "last month" resolves to literal `YYYY-MM-DD` bounds), the operator
     vocabulary (the 9 `FilterOp`s with the type subsets from `opsForField`),
     the target schema
     `{"op":"and"|"or","conditions":[{"fieldKey":string,"op":string,"value"?:string}|<group>],"unresolved"?:string}`,
     two worked examples (one date-range using `gte`+`lte`, one SELECT mapping
     a label to its option **value**), and the user query.
5. **Validate, never trust**: `JSON.parse` (strip ```json fences first);
   recursive walk with depth cap 3 (the builder's `maxDepth`); each leaf must
   have `fieldKey` ∈ catalogue keys and `op` normalised via the adapter's
   `normalizeOp` (import `{ normalizeOp, pruneGroup }` from
   `'@/app/sabcrm/[objectSlug]/record-surface-adapter'` — that module is
   deliberately React/CSS/server-only-free); for SELECT/MULTI_SELECT leaves
   map a value that matches an option **label** onto the option **value**
   (case-insensitive); drop anything invalid; finish with `pruneGroup`.
   Empty result after pruning → `{ ok:false, error: 'Could not turn that into a filter.' }`.
6. Return `{ group, unresolved }`.

## Part C — Smart-view work queues

A saved view becomes a **prioritized to-work list**: the view's filters scope
the queue, its multi-sort is the priority order, and each row gets work-state
actions (Done / Snooze / Skip) that are per-user and non-destructive.

### C1. Model — additive keys on the saved-view document

The Rust view handlers persist the flattened body verbatim
(`#[serde(flatten)]` in `CreateViewInput`/`UpdateViewInput`,
`rust/crates/sabcrm-views/src/dto.rs` lines 26–48) — the established additive
pattern (`columnWidths` already rides this way, see
`record-surface-adapter.ts` lines 336–377). The queue config is one more
additive key on the view doc:

```jsonc
// sabcrm_views document, additive key
"queue": {
  "enabled": true,
  /** Leaf condition marking a record as inherently done (e.g. status eq closed).
   *  Wire shape matches the widened filter leaf: { field, op, value? }. */
  "doneWhen": { "field": "status", "op": "eq", "value": "done" },
  /** DATE/DATE_TIME field key driving the SLA chip (overdue when < now). */
  "slaField": "dueDate",
  /** Default snooze, minutes. */
  "snoozeMinutes": 1440
}
```

Priority IS the view's existing `viewSorts` (primary first) — no new key.
`kind` stays `table`/`board` etc.; the queue is a **presentation** of the view
(`vt=queue`), so the doc's typed `ViewKind` enum is untouched.

## Risks / open questions

1. **Scheduler budget**: 20 LLM evals/tick share the 300 s budget with 5
   existing passes; if backlogs grow, split into a dedicated cron
   (`/api/cron/sabcrm-ai-fields`, same `CRON_SECRET` guard) — the pass is
   already a self-contained function, so the move is mechanical.
2. **Direct-Mongo AI writes bypass the engine** (intentional, to avoid
   `updatedAt` bumps) — search indexing / audit trails will not see AI value
   changes. Acceptable for v1; revisit if `isSearchable` objects need AI
   values in search.
3. **Billing identity drift**: the scheduler bills the *project owner* while
   manual recompute/NL filter bill the *acting user* — consistent with the
   sequences-email precedent but worth a product sign-off.
4. **Model output quality**: SELECT coercion rejects non-option replies
   (fails closed); NL filter prunes invalid leaves, which can silently narrow
   intent — the `unresolved` note mitigates but does not eliminate this.
5. **`gateway`/`ANTHROPIC`/`OPENAI` keys unset** on some environments: every
   AI surface must keep the honest "AI is not configured" path
   (`generateSabcrmText` returns `ok:false`; the UI shows the error inline).
6. **Two-store gotcha**: queue state and view patches must stay on the Rust
   path (`listViewsTw`/`updateViewTw`); the native `views.server.ts` store is
   NOT used by the record surface and must not grow queue logic.
7. **`pending` claim rows** left by a crashed tick self-heal only after the
   claim delay; a stuck `pending` longer than ~10 min indicates the pass
   died mid-LLM-call — surfaced via the scheduler report lines.
