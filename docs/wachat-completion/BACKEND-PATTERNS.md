# WaChat completion — backend & wiring cheat-sheet (hand-verified)

This is the authoritative how-to for the "make every WaChat page work" campaign.
Every execution agent MUST read this before wiring a page. Patterns verified against
the live repo on 2026-06-06.

## The two backend surfaces

1. **Rust-proxied API (`/api/v1/wachat/*`)** — `@generated by tools/api-codegen — DO NOT EDIT BY HAND`.
   These are thin proxies: `withApiV1(...)` + `rustFetchAsUser(ctx.tenantId, '/v1/wachat/...')`.
   - **Reuse** them (call from the client / from a server action) but **never hand-edit** the generated `route.ts`.
   - To add/change one you must edit `tools/api-manifest/` and run `npm run api:gen` — and the real
     logic lives in a **Rust crate we cannot recompile here**. So treat missing Rust endpoints as an
     **integration seam**, not something we build in this campaign.

2. **Next-native server actions + Mongo** — THIS is what we build at scale. Fully buildable in-repo.
   - 79/100 pages already import server actions directly from a `'use client'` page.

## Canonical server action (the pattern to replicate)

```ts
// src/app/actions/<feature>.actions.ts   (or per-page src/app/wachat/<route>/actions.ts)
'use server';

import { connectToDatabase } from '@/lib/mongodb';
import { ObjectId } from 'mongodb';
import { revalidatePath } from 'next/cache';
// import the auth/tenant helper used elsewhere (see "Session & tenant" below)

export async function listThings(/* filters */) {
  const { db } = await connectToDatabase();
  const userId = await getCurrentUserId();           // scope every query by owner/tenant
  const docs = await db.collection('wa_things')
    .find({ userId })
    .sort({ createdAt: -1 })
    .toArray();
  // serialise ObjectId/Date for the client component
  return docs.map((d) => ({ ...d, _id: d._id.toString() }));
}

export async function createThing(input: { name: string }) {
  try {
    const { db } = await connectToDatabase();
    const userId = await getCurrentUserId();
    const now = new Date();
    const res = await db.collection('wa_things').insertOne({ ...input, userId, createdAt: now, updatedAt: now });
    revalidatePath('/wachat/<route>');
    return { success: true, id: res.insertedId.toString() };
  } catch (e: any) {
    console.error('createThing failed:', e);
    return { success: false, error: e.message };
  }
}
```

Rules:
- **Collections use the `wa_` prefix** (e.g. `wa_auto_reply_rules`, `wa_contacts`, …). Match existing names from the inventory; only invent `wa_*` names when none exists.
- Always **scope by the authenticated user/tenant** — never return another tenant's data. SabNode is multi-tenant SaaS.
- Mutations return `{ success: true, ... }` or `{ success: false, error }`. Call `revalidatePath('/wachat/<route>')` after writes.
- Serialise `_id`/dates before returning to client components.

## Session & tenant resolution

Helpers live in `src/lib/auth.ts`, `src/lib/jwt.ts`, `src/lib/jwt-for-rust.ts`, `src/lib/rbac-server.ts`.
Active project / phone-number context: `src/lib/sabwa/session-context.tsx`.
**Before writing a new action, grep an existing wired action** (e.g. `src/app/actions/contact.actions.ts`,
`src/app/actions/broadcast.actions.ts`) and copy EXACTLY how it gets the user/tenant + active project.
Do not invent a new auth mechanism.

## Live WhatsApp send

- Engine HTTP client: `engineFetch(path, { json })` from `@/lib/sabwa/engine-client` (personal-WhatsApp / Baileys, port 4001).
- Cloud API send via Rust: `src/lib/rust-client/whatsapp-send.ts` and route `POST /api/v1/wachat/messages/send`.
- Interactive/catalog/CTA/location/order: routes under `src/app/api/v1/wachat/messages/*`.
- For features that ultimately require the **live Meta Graph API** with real credentials (template submission,
  ads, pay, phone registration): build the Next action + Mongo persistence + a clearly-marked call to the
  existing rust-client/engine seam; degrade gracefully when the engine returns 404 (`treatNotFoundAsEmpty`).

## Frontend (page) pattern

- `'use client'` page importing 20ui from `@/components/sabcrm/20ui` and wrapped in
  `<WachatPage breadcrumb={[...]} width=...>` from `@/app/wachat/_components/wachat-page`.
- Load data via `useEffect`+server action (or pass from a tiny server wrapper). Replace ALL hardcoded/mock
  `useState` arrays with real action calls.
- Every control wired to a real action. Use `useToast` for feedback, `useTransition`/`isPending` for loading.
- **Required UX states**: loading (Skeleton/Spinner), empty (`EmptyState`), error (toast + inline). No raw
  `<div>` soup — use 20ui `Card/Table/Button/Input/Badge/Dialog/...`.

## Verify

- Typecheck: `NODE_OPTIONS=--max-old-space-size=16384 npx tsc --noEmit` (full repo; slow — run between waves).
- Build: `npm run build` (`next build --turbo`). Expensive; run at milestone checkpoints, not per page.
- Lint: `npm run lint`.

## Definition of done (per page)

1. No mock/hardcoded data — reads real data from a server action (Mongo) or reused API.
2. Every interactive control persists to a real backend and reflects the result.
3. Loading + empty + error states present; toast feedback on mutations.
4. 20ui components inside `<WachatPage>`; no inline-style soup; dark-mode safe.
5. Typechecks clean.
