# i18n — translating the SabNode dashboard

The SabNode dashboard ships with a small, in-house i18n runtime under
`src/lib/i18n/`. There is **no** `next-intl` / `react-intl` / `i18next`
dependency — the existing primitive (`t(key, params, locale)`) plus a
thin React/Next.js layer is enough, and avoids bundler config churn
under Turbopack.

This page documents the contract so new pages can be translated
consistently.

## Quick start (client component)

```tsx
'use client';
import { useT } from '@/lib/i18n/client';

export function MyPage() {
  const { t, locale } = useT();
  return <h1>{t('mymodule.page.title')}</h1>;
}
```

## Quick start (server component / server action)

```tsx
import { getT } from '@/lib/i18n/server';

export default async function MyPage() {
  const t = await getT();
  return <h1>{t('mymodule.page.title')}</h1>;
}
```

## How the locale is resolved

`getCurrentLocale()` in `src/lib/i18n/server.ts` runs once per request
inside the dashboard layout (`src/app/dashboard/layout.tsx`) and seeds
the `LocaleProvider`. Order of precedence:

1. `?locale=` query param (preview override)
2. `locale` cookie (set by `setLocale()` on the client)
3. `user.language` on the persisted session (set via the profile
   settings page → `handleUpdateUserProfile` in `user.actions.ts`)
4. `'en'` fallback

Because the cookie write happens on `setLocale()`, switching the
language in-app updates the next RSC request without reload — but a
hard reload also picks up the change. The user's stored `language`
preference is the durable source of truth.

## Where translations live

Locale dictionaries are flat `key → string` maps at:

```
src/locales/<code>/common.json
```

Shipped locales: `en`, `es`, `fr`, `hi`, `ar`, `zh`, `ja`, `pt`.

Keys are dotted strings namespaced by feature:

```
"sabflow.connections.title": "Connections",
"sabflow.connections.modal.searchPlaceholder": "Search providers (e.g. slack, notion, postgres)…",
```

ICU-lite pluralisation and `{name}` placeholders are supported by
`src/lib/i18n/translate.ts`:

```jsonc
"common.itemsCount": "{count, plural, one {# item} other {# items}}"
```

```tsx
t('common.itemsCount', { count: 3 }); // → "3 items"
```

## Translating a new page (pattern)

The reference implementation is `src/app/dashboard/sabflow/connections/page.tsx`.

1. Identify every user-visible English string in the page.
2. Mint a dotted key under `<module>.<page>.<scope>.<name>`
   (e.g. `crm.leads.table.column.email`).
3. Add the English value to `src/locales/en/common.json`.
4. Add the same key to **every other locale's `common.json`**. If you
   don't have a real translation yet, prefix the English string with
   the locale tag (e.g. `"[hi] Connections"`) so missing translations
   are visible in the UI rather than silently falling back to English.
   Do **not** machine-translate as part of feature work — translations
   are reviewed and shipped in dedicated batches.
5. Replace the literal with `t('your.key')`. For client components,
   `const { t } = useT();`. For server components / actions,
   `const t = await getT();`.
6. For values that depend on runtime data, use `{placeholders}`:
   `t('sabflow.connections.filter.all', { count: credentials.length })`.

## RTL

`useT()` exposes `direction: 'ltr' | 'rtl'` resolved from the locale
registry (`src/lib/i18n/registry.ts`). Use it when a component needs
to flip layout (`<div dir={direction}>`). The Arabic locale ships as
RTL.

## Date / number formatting

Do **not** hardcode `'en-US'` in `Intl.*` formatters. Read the locale
from `useT()` (client) or `getCurrentLocale()` (server) and pass it
through:

```tsx
new Date(d).toLocaleString(locale, { month: 'short', day: 'numeric' });
```

For currency, see `formatCurrency` in `src/lib/i18n/formatter.ts`.

## Anti-patterns

- Don't import from `@/lib/i18n` for the React layer — it doesn't
  re-export `useT()` / `getT()` on purpose (those have runtime
  constraints — `client.tsx` pulls React, `server.ts` is
  `server-only`). Always import from `@/lib/i18n/client` or
  `@/lib/i18n/server` explicitly.
- Don't call `t()` (the raw locale-agnostic one) inside components —
  it always renders the `en` string. Use the hook / `getT()`.
- Don't add a key to only one locale file. The translate runtime
  falls back to `en`, but stale keys diverge fast — add the same key
  to every locale, with `[locale] English` as a placeholder when no
  translation exists.
- Don't store user-facing strings in `enum`s or `const` maps in TS —
  put them in JSON so they're discoverable to a translator.

## File map

| File | Purpose |
| ---- | ------- |
| `src/lib/i18n/translate.ts` | Locale-agnostic `t(key, params, locale)` + dictionary lookup |
| `src/lib/i18n/registry.ts`  | Shipped locales + direction metadata |
| `src/lib/i18n/server.ts`    | `getCurrentLocale()`, `getT()` for RSC / actions |
| `src/lib/i18n/client.tsx`   | `LocaleProvider`, `useT()`, `useLocale()` hooks |
| `src/locales/<code>/common.json` | Flat translation dictionaries |
| `src/app/dashboard/layout.tsx`   | Mounts `LocaleProvider` for every dashboard page |
| `src/app/dashboard/settings/profile/page.tsx` | UI for changing `user.language` |

## Next steps for batch translation

1. Walk every `src/app/dashboard/<module>/**/*.tsx` and replace hard
   strings with `t('<module>.<page>.<scope>.<name>')`.
2. For each new key, append the English value to `src/locales/en/common.json`
   and a `[locale] English` placeholder to every other locale file.
3. When real translations land, replace the `[hi]`-style placeholders.
   The translate runtime falls back to `en` for any unknown key, so
   partial coverage is safe to ship.
